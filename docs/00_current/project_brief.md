# Magic Watch Current Project Brief

日期：2026-05-26

## 当前定位

Magic Watch 当前不是以继续堆智能手表页面为目标。

当前阶段目标是验证一套可维护、可扩展、事件驱动、可模拟、未来可接硬件的智能手表嵌入式应用架构。

v0 的成功标准不是页面数量，而是：

1. 新增一个功能时，修改范围可预测。
2. 输入、服务、状态机、页面之间职责清楚。
3. 页面不直接触碰硬件。
4. `PageManager` 不做业务判断。
5. 真实硬件接入时，上层 UI 和状态机尽量不重写。
6. 文档入口足够短，历史资料可检索但不污染默认上下文。

## 当前阶段

当前进入“架构收口阶段”，暂停继续扩张健康、通知、设置等功能页。

本阶段优先恢复解释力：

- 项目为什么继续推进。
- 当前代码主线是什么。
- 哪些模块值得保护。
- 哪些状态和事件应该归属到哪个领域。
- 哪些文档默认读，哪些只作为历史检索。

## 当前不做

- 不推倒重写模拟器。
- 不继续盲目填充页面。
- 不把 `ShellPages.cpp`、`SettingsPages.cpp` 或 `AppStateMachine.cpp` 只做机械拆文件。
- 不在文档入口未止血前继续扩大默认阅读范围。
- 不在 Controller 中直接操作 UI、`PageManager` 或 LVGL 对象。

## 当前优先级

1. 文档入口止血。
2. `AppStateMachine` 状态与事件盘点。
3. 提取第一个真实子控制器 `PowerController`。
4. 建立 UI 生命周期契约。
5. 再拆 `SettingsPages.cpp` 和 `ShellPages.cpp`。
6. 最后补硬件接入边界。

## 本阶段完成标准

本阶段完成不以新增页面数量或文件变短为标准，而以系统解释力恢复为标准：

1. 新会话只读 `AGENTS.md`、`docs/document_map.md` 和 3-5 个当前文档即可理解项目方向。
2. 每轮开发都有明确 Scope Lock，实际 diff 不越过本轮允许文件。
3. `AppStateMachine` 的状态变量、事件路径、`PageManager` 调用点已有归属说明。
4. 第一个子控制器能通过 Action 返回策略结果，而不是直接操作 UI 或 `PageManager`。
5. 拆 UI 巨石前已有 LVGL 生命周期契约，避免 timer、订阅和对象树释放责任继续散乱。
6. 文档入口不再默认读取历史流水账或专题审计文档。
