# 硬件候选约束表 v1

日期：2026-05-21

## 目的

本文是硬件前哨阶段的第二个小闭环，用来把“应该看什么硬件”变成可比较的约束表。

本轮仍然不列具体购买短名单，不绑定最终平台，不启动 BSP 或驱动开发。所有候选硬件都应先进入同一张约束表，再决定是否值得进入下一轮规格核验。

## 使用规则

- 先评估约束，再讨论喜好。
- 先判断能否验证关键架构假设，再判断是否适合作为最终产品基础。
- 第一块验证板不需要完美，但必须能完成最小 bring-up。
- TTGO_TWatch 可以作为已有参考板进入表格，但不默认成为最终目标。
- 如果某个候选在资料、调试或供货上存在明显不确定，应先标记风险，不用强行淘汰。

评分建议：

| 分值 | 含义 |
| --- | --- |
| 0 | 不满足，或会明显阻断硬件前哨验证。 |
| 1 | 勉强满足，需要临时替代方案或额外风险接受。 |
| 2 | 满足当前阶段需要。 |
| ? | 资料不足，下一轮必须核验。 |

## 必须回答的硬约束

这些项目不要求都满分，但必须能给出明确回答。若连续多个项目为 `?`，不适合进入第一块验证板。

| 维度 | 约束问题 | 期待答案 |
| --- | --- | --- |
| 显示 | 能否点亮屏幕并运行最小 LVGL 画面 | 必须能做到。 |
| 显示 | 屏幕尺寸和安全区是否能承载当前 240 x 296 方向的 UI 假设 | 至少能用于验证主要布局压力。 |
| 触摸 | 是否有触摸输入，或是否有足够的替代输入 | 第一块板可替代，但需记录缺口。 |
| 触摸唤醒 | 触摸控制器是否支持低功耗中断 / 唤醒 | 不必立即实现，但必须能查清。 |
| 物理输入 | 是否有按键、表冠、旋钮或可替代 `CrownPress` 的输入 | 第一块板必须具备至少一种。 |
| 姿态 | 是否有 IMU，或是否能外接 / 模拟姿态输入 | 第一块板最好具备；没有则只能验证部分路径。 |
| IMU 唤醒 | IMU 是否支持低功耗运动 / 姿态中断 | 必须能查清，否则无法评估 `RaiseToWake`。 |
| 电池 | 是否能读取电压、电量、充电或外部供电状态 | 至少应能验证一种电池状态来源。 |
| 电源 | sleep / deep sleep 后哪些输入仍可唤醒 | 必须能实验或从资料确认。 |
| 调试 | 是否有稳定串口日志、烧录和断点调试路径 | 第一块板必须满足。 |
| 生态 | 是否有 LVGL、屏幕、触摸、IMU 示例或驱动资料 | 资料越成熟，越适合第一块验证。 |

## 评分维度

### 显示与 LVGL

关注点：

- 屏幕分辨率、比例、圆角、安全区。
- LVGL 移植路径是否清楚。
- 显存、帧缓冲和刷新性能是否可承受。
- 是否支持低刷新、局部刷新或低功耗显示模式。
- 是否能验证 `ScreenOffPage` 与 AOD / 息屏显示假设。

最低要求：

- 第一块验证板必须能运行最小 LVGL 画面。
- 如果屏幕形态和当前模拟器差异很大，必须记录它验证的是“硬件能力”而不是“最终 UI 适配”。

### 触摸与表冠 / 按键

关注点：

- 触摸点击、拖动、滑动是否容易接入。
- 触摸控制器是否支持中断。
- 是否支持低功耗触摸唤醒。
- 是否有真实表冠 / 编码器。
- 如果没有表冠，是否至少有按键可映射为 `CrownPress`。

最低要求：

- 第一块验证板必须有一种可稳定触发 wake / press 路径的物理输入。
- 表冠旋转不是第一块板的硬要求，但必须记录是否能验证 `CrownSample::RotateCW / RotateCCW`。

### IMU 与姿态

关注点：

- 是否板载 IMU。
- 是否支持中断唤醒。
- 是否支持低功耗姿态、敲击、抬腕或运动检测。
- SDK 示例是否能读出稳定姿态数据。
- 是否能把硬件事件映射到当前 `RaiseToWake` 验证路径。

最低要求：

- 若第一块板没有 IMU，必须说明本轮不能验证真实 `RaiseToWake` 功耗路径。
- 如果有 IMU，但没有低功耗中断资料，只能算“姿态数据验证”，不能算“翻腕唤醒验证”。

### 电源、电池与低功耗

关注点：

- 电池电压、电量百分比、充电状态、外部供电状态来源。
- PMIC 或电源管理能力。
- 主控 sleep / deep sleep 后外设保持情况。
- 屏幕关闭和系统睡眠是否是两个不同层次。
- 功耗测量是否方便。

最低要求：

- 第一块验证板至少要能观察 screen off / wake 的真实限制。
- 不能把“屏幕变黑”直接等同于真实低功耗。

### 主控、内存与生态

关注点：

- RAM / Flash 是否足够运行 LVGL 和基础资源。
- SDK、编译链、烧录工具、调试器是否稳定。
- 是否容易接入日志。
- 是否有成熟示例工程。
- 是否容易扩展到后续 RTOS / 服务层实验。

最低要求：

- 第一块验证板应优先选择资料成熟、bring-up 风险低的平台。
- 不因低功耗潜力高就选择资料极少、调试困难的平台作为第一块验证板。

## 候选分类

下一轮列具体候选时，先按类别放入表格。

| 类别 | 用途 | 优点 | 风险 |
| --- | --- | --- | --- |
| 已有参考板 | 复用手头硬件，快速验证屏幕、输入、传感器路径 | 成本低，启动快 | 可能被既有板子能力反向绑定 |
| ESP32 类快速验证板 | 快速跑通 LVGL、触摸、IMU、Wi-Fi/BLE 生态 | 资料多，bring-up 快 | 低功耗路线可能不是最终目标 |
| 低功耗 MCU / RTOS 候选 | 验证更接近最终低功耗架构的主控路线 | 更贴近长期目标 | UI、屏幕和生态 bring-up 成本可能高 |
| 高集成手表开发板 | 同时验证屏幕、触摸、IMU、电池和外壳形态 | 接近真实产品 | 平台绑定风险高，资料质量差异大 |
| 显示 / 触摸模组 | 单独验证屏幕形态、触摸控制器和 LVGL 压力 | 边界清晰 | 不能验证完整电源和姿态路径 |

## 第一块验证板选择规则

第一块验证板的目标是“降低未知”，不是“证明最终方案正确”。

优先选择满足以下条件的硬件：

- 能在较短时间内点亮屏幕并跑最小 LVGL。
- 至少有一种可靠物理输入可映射到 `CrownPress`。
- 有触摸或可替代的输入路径。
- 有 IMU，或能方便外接 / 模拟姿态输入。
- 能读取至少一种电池或供电状态。
- 能实验 screen off / wake 后哪些状态保留、哪些状态丢失。
- 资料、示例、烧录、日志和调试链路足够成熟。

第一块验证板可以暂时不满足：

- 最终外观形态。
- 最终功耗目标。
- 真实表冠旋转。
- 完整 AOD / 息屏显示。
- 完整触摸边缘手势。
- 完整应用服务。

如果两个候选都能完成最小 bring-up，优先选择调试更容易、资料更成熟、失败成本更低的候选。
