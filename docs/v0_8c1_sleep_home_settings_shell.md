# v0.8c-1 Sleep 首页与设置壳层闭环

日期：2026-05-24

## 本轮目标

把 `AppSleep` 从 placeholder 页面推进为一条真实可达的应用路径，同时只收下面这一个小闭环：

- `Sleep` 首页不再是占位页
- 首页具备：
  - 昨晚睡眠记录占位卡
  - 近 7 天睡眠占位卡
  - `设置` 入口
  - `说明` 入口
- `设置` 页接入当前已经存在的共享健康监测设置模型
- `说明` 页先保留壳层，正文留到下一轮

## 为什么不一轮把 Sleep 全做完

`Sleep` 如果一轮同时做完首页、近 7 天图表、昨晚详情、设置编辑、说明正文和睡眠数据模型，会同时碰：

- 页面结构
- 共享设置模型
- 睡眠数据模型
- 图表排布
- 多级导航
- 回归验证

这会让上下文和回归面迅速膨胀。  
因此本轮只先解决：

1. `AppSleep` 真实可达
2. Sleep 设置可以读取和编辑两项当前最相关的共享设置
3. 说明页入口和壳层成立

## 本轮范围

### 1. Sleep 首页

- 顶部标题与时间
- 昨晚睡眠记录占位卡
- 近 7 天睡眠占位卡
- `设置` 卡
- `说明` 卡

说明：

- `昨晚睡眠记录` 与 `近 7 天睡眠` 当前只承担页面结构职责
- 本轮不引入真实睡眠数据模型

### 2. Sleep 设置页

本轮只开放两项和 Sleep 自身最直接相关的设置：

- `睡眠高精度监测`
- `睡眠呼吸质量监测`

结构为两层：

- `Sleep 设置` 列表页
- 每项一个详情页，详情页内包含：
  - 开关
  - 当前项的简要说明

### 3. Sleep 说明页

- 本轮只保留入口和页面壳层
- 说明正文下一轮补齐

## 数据边界

本轮不新增新的 Sleep service。

Sleep 设置详情页直接消费并写回现有共享模型：

- `HealthMonitoringSettingsModel.high_precision_sleep_enabled`
- `HealthMonitoringSettingsModel.sleep_breathing_quality_enabled`

因此本轮的系统关系是：

`BatteryOptimizationPage` 与 `Sleep` 设置页  
共同编辑同一份 `HealthMonitoringSettingsModel`

## 验收标准

1. `AppSleep` 不再是 placeholder
2. 首页可以进入 `设置` 与 `说明`
3. `设置` 页可以进入两个详情页
4. 两个详情页中的开关可编辑
5. 从 `Sleep` 设置改动后，再回到 `设置 -> 电池 -> 续航优化`，两边状态一致
6. 本轮不破坏通知、长续航、电池路径
