## 2026-05-25: 把中文编码规则从“全局停机”收成“当前文件熔断”，并确认 PowerShell 7 仅能手动进入

背景：
- 上一轮编码事故之后，项目把“看到疑似乱码就立即暂停当前功能开发”写成了强规则。
- 这条规则在安全性上有效，但在实际执行中容易被放大成“只要碰中文文档就整体放慢”，明显拖慢了当前小闭环推进。
- 进一步排查确认：PowerShell 7 已成功安装，Codex 终端里输入 `pwsh` 可以进入 `PowerShell 7.6.2`，但当前内置 shell 默认仍是 `Windows PowerShell 5.1`，不能通过下拉直接切成 `pwsh`。

决定：
- 把编码处理规则从“全局停机”收成“当前文件熔断”：
  - 无关输出里的疑似乱码，不再导致整轮功能开发暂停
  - 只有当前要修改的中文文件受到影响时，才暂停该文件相关开发并先做编码修复闭环
- 在 `docs/workflow.md` 中同步更新这条规则。
- 把 `pwsh` 明确为处理中文文档时的推荐执行环境，而不是硬依赖。

理由：
- 真正需要保护的是“不要基于乱码输出改写当前文件”，不是让整个项目一遇到中文就全局降速。
- 当前 Codex 环境下 `pwsh` 可手动使用但不是默认 shell，把它写成推荐项更符合真实执行环境。
- 这样能同时保住编码安全和小闭环推进效率。

复盘点：
- 后续如果中文相关任务明显变多，优先在终端里手动进入 `pwsh` 后再做检查和验证。
- 只要当前要修改的中文文件本身正常，就不要因为其他无关输出异常而整轮停住。

## 2026-05-24: 中文文档一旦出现疑似乱码，必须先停下来做编码修复闭环

背景：
- 在本轮推进中，`docs/simulator_manual_regression_matrix.md` 出现了真实的中文 mojibake。
- 问题不是简单的“显示乱码”，而是错误解码后的乱码文本被重新按 UTF-8 写回文件。
- 如果继续基于乱码终端输出 patch 文档，会把损坏范围继续扩大。

决定：
- 把编码问题视为独立事故处理，而不是普通文档小问题。
- 一旦中文文档在终端、diff 或脚本输出里出现疑似乱码，必须立即暂停当前功能开发。
- 先区分“只是显示问题”还是“文件内容已损坏”，确认后只做编码修复小闭环。
- 在 `docs/workflow.md` 中补充最小流程约束：编码问题修复前，不得基于乱码输出继续 patch 文件。

理由：
- 中文文档是项目长期记忆，编码损坏会直接污染需求、决策、回归矩阵和后续会话理解。
- 这类问题越晚处理，恢复成本越高，也越容易把错误文本当成真实内容继续传播。

复盘点：
- 修改中文文档后，应补一次乱码哨兵检查。
- 只要看到 `锛`、`鏃`、`褰`、`闈`、`€`、`�` 等疑似串，就先停，不继续顺手开发。

## 2026-05-24: 修正长续航模式屏蔽通知能力

背景：
- 在通知阶段封箱回归中发现：长续航模式熄屏状态下收到消息，仍然可以被通知唤醒，并且进入消息页。
- 从消息详情页通过底部把手退出时，还会短暂露出普通表盘，再回到长续航页面。
- 这不是单纯的过场问题，而是规则错误：长续航模式按产品约束只保留时间、计步、NFC，不应保留通知能力。

决定：
- 在 `AppStateMachine` 中增加长续航模式的通知屏蔽规则。
- 长续航模式下直接忽略 `NotificationWakeRequested`。
- 同时禁止 `Notifications` 和 `NotificationWakePreview` 这两个壳层在长续航模式下被打开。

理由：
- 仅修“退出时不要闪普通表盘”会掩盖问题本质，通知能力本身就不应该在长续航模式下存在。
- 把规则收在状态机入口层，比在单个页面里打补丁更稳，也更不容易被后续通知功能绕开。

复盘点：
- 后续任何新增通知交互，都必须默认经过“长续航模式是否允许”的检查。
- 长续航回归不应只测表盘和退出页，也要包含“通知被彻底屏蔽”的约束。
