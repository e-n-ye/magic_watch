# Magic Watch Decision Log

## 2026-05-21: 进入硬件前哨验证 / 选型约束收敛阶段

背景：
- v0.2 shell 已经形成可解释、可回归的模拟器基线。
- 表盘中心交互、Launcher、Quick Settings、Settings、screen off / wake 路径已经形成主线闭环。
- 继续扩展 Weather、Steps、Sleep、Health、NFC、支付等 mock 页面，除非服务于硬件验证，否则收益开始下降。
- 当前项目仍未完成最终硬件选型，也不应过早绑定 TTGO_TWatch、ESP32 或任何单一平台。

决定：
- 进入“硬件前哨验证 / 选型约束收敛”阶段。
- 新增 `docs/hardware_frontier_entry.md` 作为本阶段入口文档。
- 第一轮采用约束优先，不列具体候选硬件短名单。
- TTGO_TWatch 只作为已有参考硬件或验证板，不作为默认最终目标。

理由：
- 真实硬件能够反向验证当前模拟器阶段的关键假设，包括熄屏、亮屏、触摸、表冠 / 按键、IMU、电池和低功耗显示。
- 当前 `HAL::Device`、`BatterySample`、`CrownSample`、`TouchSample`、`MotionSample` 已经提供了最小评审对象。
- 先定义约束和门槛，可以避免项目从“模拟器堆页面”直接跳到“被某块开发板牵着走”。

边界：
- 本阶段不做最终硬件选型。
- 本阶段不启动完整 BSP、RTOS 任务划分或驱动体系建设。
- 本阶段不承诺真实功耗水平。
- 本阶段不实现真实天气、健康、通知、支付、NFC 等业务服务。
- 只有完成至少一块验证板的最小 bring-up，并识别出模拟器语义与真实平台能力的差异后，才考虑进入正式硬件适配分支。

复盘点：
- 后续检查硬件前哨文档是否真的帮助缩小选型空间，而不是变成新的愿望清单。
- 当完成第一块验证板 bring-up 后，复盘 `TapToWake`、`RaiseToWake`、AOD / 息屏显示和页面恢复假设是否仍然成立。

## 2026-05-14: Web 视觉沙盘作为 LVGL 之前的视觉规格中间层

背景：
- 当前直接从参考图和文字描述跳到 LVGL 模拟器代码，容易在小屏表面上丢失比例、文字预算和层级意图。
- `HomeShortcutPage` 的通用卡片模板已经暴露出明显局限：信息密度偏高、说明文字残留、视觉反馈成本高。
- 项目最终仍然要落回 `LVGL` 与单片机部署，不能为了网页稿好看而引入难以回译的视觉依赖。

决定：
- 新增 Web 视觉沙盘作为设计中间层。
- Web 沙盘只负责视觉规格验证，不替代 LVGL，也不作为自动代码源。
- 第一块沙盘只做天气快捷页，用于锁定安全区、卡片比例、文字预算和图标槽位。
- Web 沙盘必须遵守 `LVGL` 友好约束，只使用纯色块、圆角、透明度、位图图标和简单层级。

理由：
- 它比“直接改 LVGL 再截图返工”拥有更高的视觉反馈效率。
- 它比“参考图 + 聊天描述 -> 凭感觉写代码”多了一层可版本管理的设计契约。
- 它比“网页自动转 LVGL”成本更低、风险更小，也更适合当前项目阶段。

复盘点：

- 后续检查：Web 沙盘是否真的降低了 LVGL 回译偏差，而不是变成另一套孤立原型。

## 2026-05-13: Phase 0 先建立项目宪章和协作规范

背景：

- 项目目标很大：智能手表、嵌入式架构、低功耗、C++、事件驱动、长期 AI 协作。
- 当前 `magic_watch` 只有一个 LVGL PC 模拟器工程，并夹杂之前开发残留。
- 过去项目多次遇到复杂后难维护的问题。

决定：

- 第一阶段不重写业务代码。
- 先建立 Git 仓库内文档：项目宪章、协作流程、学习计划、架构原则、决策记录。
- 让未来会话能通过文档恢复上下文。

理由：

- 如果没有共同工作范式，后续会反复在“重写、换框架、堆功能”之间摇摆。
- 文档化边界能降低长期项目失忆风险。
- 对学习型项目来说，记录为什么这么做和做了什么同样重要。

复盘点：

- 后续每个阶段结束时检查：文档是否真的帮助恢复上下文，还是变成了负担。

- 现有模拟器代码已经能支撑轻量事件流实验。
- 等复杂度证明需要框架时，再带着问题引入。

复盘点：

- 如果后续出现多个独立服务互相耦合、状态机膨胀、事件顺序难以保证，再重新评估 QPC。

## 2026-05-13: 支付、NFC、消息、健康等在当前阶段先做交互壳与 mock 数据

背景：

- 用户希望表盘周围快速入口承载支付、NFC、健康、天气、睡眠、音乐等内容。
- 这些功能一旦追求真实可用，将迅速牵涉蓝牙、手机侧协议、传感器、算法、存储和硬件能力。

决定：

- mock 数据模型未来更容易替换成真实服务。
- 这更符合项目宪章中“小步前进、明确边界、先能解释系统”的原则。

复盘点：

- 当主页闭环稳定后，再按优先级选择一个真实领域服务替换 mock 数据，例如通知、天气或计步。

## 2026-05-13: 熄屏/亮屏策略作为独立设计主题处理

背景：

- 用户补充了完整的显示策略需求：表冠按下亮屏、翻腕亮屏、单击亮屏、超时熄屏、姿态熄屏、遮盖熄屏、持续亮屏。
- 这些行为既影响交互，也影响未来低功耗设计。
- 其中 `TapToWake`、`RaiseToWake`、`AlwaysOnDisplay` 的真实效果明显依赖硬件能力。

决定：

- 将熄屏/亮屏策略单独建模，而不是只把它视作“按一下就黑屏/亮屏”。
- 当前阶段只在模拟器中定义策略、配置项和状态切换规则。
- 长按表冠功能先预留，不绑定关机，也不在当前阶段固定为某个动作。

理由：

- 显示策略是未来低功耗和输入系统之间的重要边界，值得单独拥有状态和配置。
- 先定义策略层，后续更容易迁移到真实触控控制器、IMU 和显示驱动。
- 这能避免在硬件未定时对功耗表现做过度承诺。

复盘点：

- 当硬件方向明确后，重新评估 `TapToWake`、翻腕亮屏和持续亮屏的真实可行性与功耗成本。

## 2026-05-13: 左边缘右滑手势在主页环与普通页面中采用不同语义

背景：

- 用户补充说明：左边缘右滑在表盘和左右 4 个快速入口页中，不应作为通用返回，而应继续遵循主页环流转逻辑。
- 在应用页、消息页、快捷设置页等其他页面中，同一手势应返回上级页面。

决定：

- 将左边缘右滑定义为上下文敏感手势。
- 在主页环中，它属于主页流转意图。
- 在非主页环页面中，它属于标准返回意图。

理由：

- 主页环本质上是横向浏览空间，不是普通层级页面栈。
- 这样能保持主页体验连贯，同时保留手表常见的边缘返回操作。
- 如果不在意图层提前分流，后续状态机会很容易被手势例外规则污染。

复盘点：

- 实现样机后观察：消息页、快捷设置页等壳层页面是否都应统一走返回上级，还是个别页面需要更细分的父级定义。

## 2026-05-13: 流式页面的滚动需要明确物理反馈规则

背景：

- 用户补充说明：在消息页、设置页、应用页等流式页面中，用户既可以通过触屏上下滑动浏览，也可以通过表冠旋转浏览。
- 触摸滚动应具备跟手、惯性、减速和缓冲，快速滑动后松手，内容需要继续滑动一小段再停下。

决定：

- 将滚动物理反馈作为独立交互规则处理。
- 当前阶段先定义触摸拖动、快速滑动、惯性衰减和表冠滚动的一致性原则。
- 暂不在文档阶段把具体阻尼、加速度、减速度参数写死。

理由：

- 小屏设备对滚动手感非常敏感，手感差会直接破坏产品质感。
- 这部分属于系统交互体验，不应散落在单个页面里临时处理。
- 先抽象出统一规则，后续更容易在 LVGL 模拟器和真实硬件上对齐。

复盘点：

- 样机阶段重点观察：触摸滚动是否足够跟手，惯性是否自然，表冠滚动是否与触摸体验割裂。

## 2026-05-22: T-Watch S3 Plus 第二闭环采用 LVGL 指针事件观测触摸

背景：

- 第一闭环已经验证最小显示、串口日志和 LVGL 页面刷新。
- 官方 `LV_Helper` 已把 FT6336U 触摸作为 LVGL pointer 输入设备注册。
- 本阶段目标是硬件前哨验证，不应提前引入复杂手势或 Magic Watch 页面状态机。

决定：

- 第二闭环只在 `prototypes/twatch_s3_plus_bringup` 中扩展基础触摸观测。
- 使用 LVGL 输入设备读取按下、拖动、释放和坐标，不重写底层 FT6336U 驱动。
- 屏幕显示最近触摸状态、坐标和事件计数；串口输出 `[bringup-touch] press/move/release`。
- BMA423、AXP2101、screen off / wake 继续留给后续独立闭环。

理由：

- 复用官方触摸接入路径能降低硬件 bring-up 摩擦。
- 触摸坐标方向和边缘可靠性可通过最小 UI 与串口日志独立验收。
- 保持 T-Watch S3 Plus 作为参考验证板，不把它绑定为最终触摸方案。

复盘点：

- 2026-05-22 实物观察确认：按下、释放和事件计数有效，左上约为 `0,0`，右下约为 `240,240`，当前 `rotation=2` 下坐标方向与视觉方向一致。
- 后续若要评估边缘可靠性，再增加多次点击统计或原始 FT6336U 坐标记录。

## 2026-05-22: T-Watch S3 Plus 第三闭环只读取 AXP2101 基础供电状态

背景：

- 第一闭环已经验证显示和日志，第二闭环已经验证 FT6336U 基础触摸。
- 官方 `PMU_ADC` 示例提供了 AXP2101 基础状态读取路径。
- `watch.begin()` 已启用电池检测、VBUS 电压、电池电压和系统电压测量。

决定：

- 第三闭环只在 `prototypes/twatch_s3_plus_bringup` 中加入 AXP2101 基础读数观察。
- 页面显示 USB、充电、放电、充电阶段、电池电压、电量百分比和系统电压摘要。
- 串口输出 `[bringup-pmu]`，每两秒记录一次 USB、充电、放电、电池/VBUS/SYS 电压和电量。
- 不处理 PMU 中断、PEK 按键、供电轨配置、充电策略、关机策略或 sleep/wake。

理由：

- 当前目标是硬件前哨验证，先确认读数链路和状态变化可观察。
- 供电策略和唤醒策略会影响系统行为，必须独立成后续闭环，避免在一个实验里混入太多变量。
- T-Watch S3 Plus 仍只作为参考板，不把 AXP2101 绑定为最终 PMIC 方案。

复盘点：

- 2026-05-22 串口观察确认 USB 接入充电状态：`usb=Y`、`charging=Y`、`discharging=N`、`chg=cc`，电池约 `3540mV`，VBUS 约 `4758mV`，SYS 约 `4747mV`，电量约 `4%`。
- 2026-05-22 实物拔插确认：拔掉 USB 后显示 `usb=N`、`charging=N`、`discharging=Y`，插回 USB 后恢复为 USB 接入充电状态。
