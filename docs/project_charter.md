# Magic Watch Project Charter

## Project Identity

Magic Watch 是一个以智能手表为载体的长期学习型嵌入式项目。它的目标不是做一个短期 Demo，而是帮助项目作者完成从“前后台轮询/多个 RTOS 循环”到“自顶而下、事件驱动、可维护系统架构”的转变。

目标参照包括 InfiniTime、X-TRACK、可穿戴系统岗位要求，以及已有手表项目经验。参照不等于复制：本项目会从小系统开始，逐步沉淀自己的架构、文档和工程节奏。

## Long-Term Goals

- 做出一个具备真实智能手表完整度的系统：表盘、启动流程、页面管理、传感器、功耗管理、存储、调试、OTA 或升级机制等。
- 训练企业级嵌入式软件能力：模块边界、接口设计、状态机、事件流、可测试性、文档化决策。
- 形成可持续 AI 协作范式：新会话能通过仓库文档恢复上下文，避免项目长期推进时失忆。
- 建立低功耗意识：硬件未定时也要把功耗状态、唤醒源、服务生命周期纳入架构约束。

## Phase 0 Scope

当前阶段只做项目启动和预研骨架，不追求完整产品。

本阶段包含：

- 初始化 Git 仓库。
- 建立项目宪章、协作流程、学习计划、架构原则、决策记录。
- 保留现有 PC LVGL 模拟器作为预研资产。
- 明确 v0 以模拟器优先，不绑定 TTGO_TWatch 或 ESP32。

本阶段不包含：

- 重写模拟器工程。
- 清理历史残留。
- 最终硬件选型。
- 引入完整 QPC 框架。
- 开始大规模业务功能开发。

## Success Criteria

Phase 0 完成时，应满足：

- 新会话只读 `AGENTS.md` 和 `docs/` 入口文档，就能理解项目目标、工作节奏和当前默认技术路线。
- 用户和 AI 对第一阶段的边界一致：先建长期协作基础，再逐步进入架构和代码。
- 关键决策有记录，未来可以复盘为什么当时这么选。
- 现有 `sim/lv_port_pc_vscode` 没有被破坏。

## Project Values

- 小步前进胜过一次性完美。
- 明确边界胜过堆功能。
- 可复盘的决策胜过即时灵感。
- 数据流和状态边界胜过到处互相调用。
- 先能解释系统，再扩展系统。
