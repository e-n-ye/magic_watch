# 硬件前哨阶段入口

日期：2026-05-21

## 阶段定位

当前项目进入“硬件前哨验证 / 选型约束收敛”阶段。

这个阶段不是最终硬件选型，也不是正式硬件适配分支。它的目标是用少量真实硬件验证当前模拟器阶段沉淀出的架构假设，尤其是输入、显示、熄屏、亮屏、状态恢复和低功耗相关假设。

本阶段的核心问题不是“买哪块板子做最终产品”，而是：

- 当前 `HAL::Device` 抽象是否足够承载真实硬件输入和数据。
- 当前熄屏 / 亮屏策略在真实硬件上需要哪些底层能力。
- 当前 UI 尺寸、安全区和 LVGL 渲染假设是否会被真实屏幕反向约束。
- 当前 `TapToWake`、`RaiseToWake`、AOD / 息屏显示等产品语义是否有可接受的硬件实现条件。

## 当前依据

项目已经具备进入硬件前哨验证的最低条件：

- v0.2 shell 已形成可解释、可回归的模拟器基线。
- 表盘中心交互、Launcher、Quick Settings、Settings、screen off / wake 路径已经形成主线闭环。
- 显示策略已经从页面局部状态中抽出为 `DisplayPolicyModel` 和 `DisplayPolicyRules`。
- `HAL::Device`、`BatterySample`、`CrownSample`、`TouchSample`、`MotionSample` 等硬件侧事件抽象已经存在。
- `docs/simulator_manual_regression_matrix.md` 已经记录模拟器主路径回归项。

因此，继续扩展 Weather、Steps、Sleep、Health、NFC、支付等 mock 页面，除非服务于硬件验证，否则当前收益开始下降。下一阶段更有价值的工作是让真实硬件反向检验这些模拟器假设。

## 本阶段不做什么

为了避免过早被具体板子和驱动细节牵引，本阶段明确不做：

- 不绑定 TTGO_TWatch、ESP32 或任何最终硬件平台。
- 不承诺真实功耗水平。
- 不启动完整 BSP、RTOS 任务划分或驱动体系建设。
- 不实现真实天气、健康、通知、支付、NFC 等业务服务。
- 不把当前模拟器 HAL 类型宣布为最终硬件接口。
- 不因为发现硬件限制而立即大改模拟器 UI 或状态机。

TTGO_TWatch 可以作为已有参考硬件或验证板，但它不是默认最终目标。

## 选型约束

后续筛选候选硬件时，应先按约束评估，而不是先按喜好选板。

### 显示与 LVGL

- 屏幕尺寸、比例、圆角和安全区是否接近当前模拟器假设。
- 分辨率是否足以承载当前表盘、Launcher、Settings 和 screen-off 样式。
- 显示接口、刷新方式和缓冲区需求是否适合 LVGL。
- 是否支持局部刷新、低刷新率或其他低功耗显示路径。
- 当前 `240 x 296` 圆角矩形方向是否仍然是合理的主要设计靶心。

### 触摸与输入

- 触摸控制器是否支持中断唤醒。
- 是否支持低功耗触摸扫描，决定 `TapToWake` 是否真实可行。
- 触摸事件是否能稳定映射为点击、拖动、边缘手势和滑动。
- 是否存在物理按键、旋钮、编码器或可替代表冠的输入。
- 表冠旋转是否能产生稳定方向和 detent 计数。

### IMU 与姿态

- IMU 是否支持低功耗姿态检测或运动中断。
- 是否能在主控睡眠时触发翻腕亮屏路径。
- 是否能区分 raise-to-wake 和 raise-dismiss 这类产品语义。
- 是否需要先用模拟姿态输入替代真实算法。

### 电源与电池

- 是否有电池电量计，或只能通过 ADC 估算电量。
- 是否能读取充电状态和外部供电状态。
- PMIC 或电源路径是否支持屏幕、触摸、IMU 分域管理。
- 主控 sleep / deep sleep 后哪些外设还能作为唤醒源。
- 调试状态下的功耗观测是否足够可靠。

### 主控与生态

- RAM / Flash 是否足以承载 LVGL、图片资源、页面缓存和后续服务。
- SDK、编译链、调试器和示例工程是否稳定。
- 是否容易接入串口日志、断点调试和功耗测量。
- LVGL 移植样例、屏幕驱动、触摸驱动、IMU 驱动资料是否可获得。
- 社区资料是否足以降低 bring-up 风险。

## 第一块验证板的最小 bring-up 清单

第一块验证板只验证硬件前哨问题，不追求完整产品。

最小闭环应包括：

- 点亮屏幕并运行一个最小 LVGL 画面。
- 产生等价 `CrownPress` 的物理输入事件，可以来自表冠、按键或临时替代输入。
- 触摸能产生基础点击和滑动事件。
- IMU 或模拟姿态输入能进入 raise-to-wake 验证路径。
- 电池 / 充电状态能映射到等价 `BatterySample` 数据。
- 能观察真实 screen off / wake 限制，例如屏幕关闭后哪些输入仍可唤醒。
- 能记录一次从 screen off 到 wake 再恢复 UI 的实验结果。

这个 bring-up 不要求：

- 完整页面移植。
- 完整触摸手势识别。
- 完整表冠旋转体验。
- 真实天气、计步、通知或健康服务。
- 最终低功耗参数。

## 模拟器假设核验表

硬件前哨阶段应重点核验以下模拟器假设。

| 模拟器假设 | 需要硬件回答的问题 |
| --- | --- |
| `TapToWake` 只是策略开关 | 触摸控制器是否真的支持低功耗唤醒扫描。 |
| `RaiseToWake` 可由 `R` 模拟 | IMU 是否能在低功耗状态下产生可靠中断。 |
| `SimCoverSleep` 可由 `C` 模拟 | 真实遮盖检测依赖触摸、光感、距离还是软件策略。 |
| AOD / 息屏显示可由 `ScreenOffPage` 渲染 | 屏幕和驱动是否支持低功耗静态显示，代价是多少。 |
| auto screen off 是状态机切换 | 真实硬件需要关闭哪些电源域，恢复时哪些状态会丢失。 |
| wake 后可恢复原页面 | 睡眠深度、RAM 保持和页面资源生命周期是否允许这种恢复。 |
| `BatterySample` 足够表达电量 | 真实硬件是否能提供电量、电压、充电和外部供电状态。 |
| `CrownSample` 足够表达表冠 | 候选硬件是否有真实旋转输入，或者需要按键替代策略。 |

## 进入正式硬件适配分支的门槛

只有满足以下条件后，才考虑从硬件前哨进入正式硬件适配分支：

- 至少一块验证板完成最小 bring-up。
- 核心输入可以映射到现有 HAL 事件模型，或已明确记录需要扩展的事件边界。
- 已识别哪些模拟器语义只是调试替代，哪些可以成为真实平台接口。
- 已有硬件候选对比依据，包括显示、触摸、IMU、电源、调试和生态。
- 已明确真实硬件对 `TapToWake`、`RaiseToWake`、AOD / 息屏显示的能力边界。
- 已记录一次硬件实验结果，并能反向更新模拟器阶段的假设。
