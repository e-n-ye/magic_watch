# v0.7c-2a 通知清空语义收口

日期：2026-05-24

## 目标

本轮只做一件事：

- 把 `Notifications` 列表页的 `清空` 从“直接执行”改成“先确认，再清空”

本轮不做：

- 只清已读 / 保留未读
- 独立通知详情页
- 通知动作按钮
- 已读筛选或分组

## 为什么先做这个

当前通知路径已经具备：

- 通知上游注入
- 通知预览与壳层联动
- 已读 / 未读与列表详情入口

但 `清空` 按钮仍然是：

```text
点击
-> DataCenter::clear_notifications()
```

这意味着：

- 误触代价很高
- 用户没有最后一次确认机会
- 它比“详情页是否独立”更接近真实使用中的风险点

所以本轮先把最容易误操作的地方压稳。

## 当前前后对比

### 改之前

```text
点击清空
-> 立即清空全部通知
```

### 改之后

```text
点击清空
-> 弹确认层
-> 取消：返回列表，不改数据
-> 确认：清空全部通知
```

## 本轮设计决定

### 1. 当前仍保留“清空全部”语义

本轮不重定义“只清已读”或“保留重要通知”。

原因是那会立刻引入新的产品规则讨论，而当前更高价值的是先把误触风险降下来。

所以本轮只确认：

- `清空` 仍代表清空当前通知列表中的全部通知
- 但必须先经过一次确认

### 2. 确认层只在列表态出现

本轮的 `清空` 入口仍只保留在通知列表态：

- 详情态不显示 `清空`
- 不在详情页里再额外扩展“删除单条”之类动作

### 3. 确认层出现时拦住底层滚动

为了避免确认层出现时，底下的通知列表还会继续响应表冠或滚动输入：

- 确认层激活期间，不再处理列表滚动和壳层拖拽

这样交互更稳定，也更接近真实系统行为。

## 修改范围

### 页面层

- `NotificationsPage`
  - 新增清空确认层
  - 新增取消 / 确认两个按钮
  - 确认层激活期间拦住列表滚动与壳层拖拽

### 保持不变

- `DataCenter::clear_notifications()` 语义
- `NotificationService`
- `NotificationWakePreview`
- 已读 / 未读规则
- 低电量与长续航路径

## 验收标准

### N1. 清空先确认

- 打开 `Notifications`
- 点击 `清空`
- 应出现确认层，而不是直接清空

### N2. 取消不改数据

- 在确认层点击 `取消`
- 返回通知列表
- 原有通知应保持不变

### N3. 确认后清空全部

- 再次点击 `清空`
- 在确认层点击 `确认`
- 通知列表应清空并进入空态

### N4. 确认层期间底层不乱动

- 让确认层显示时尝试滚动或旋转表冠
- 底层通知列表不应继续滚动

### N5. 旧路径不坏

- 清空后重新按 `N`
- 再按 `B`
- 通知注入、低电量通知和长续航主路径都不应被破坏

## 下一轮边界

如果这轮通过，下一轮再单独讨论：

- `清空全部` 是否要改成“只清已读”
- 是否需要删除单条通知
- 通知详情是否要独立成普通页面

先不要把这些内容和本轮混在一起。
