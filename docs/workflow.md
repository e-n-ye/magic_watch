# Magic Watch Workflow

## Session Start

每次新会话先做三件事：

1. 阅读 `AGENTS.md`。
2. 阅读和当前任务相关的 `docs/` 文档。
3. 用非破坏性命令观察当前 Git 状态和相关文件。

推荐命令：

```powershell
git status --short --branch
rg --files
```

## Standard Loop

每次推进遵循：

1. 观察现状：先看代码、文档、目录结构，不凭印象改。
2. 明确意图：说清本次要解决什么，不解决什么。
3. 识别分歧：如果用户或 AI 的判断可能有问题，先搜集证据再讨论。
4. 小步实施：优先做能形成闭环的小改动。
5. 验证结果：运行能覆盖本次改动的检查。
6. 记录记忆：必要时更新 `docs/decision_log.md` 或相关设计文档。

## Planning Rules

进入一个新阶段前，必须先明确：

- 目标。
- 边界。
- 验收标准。
- 主要风险。
- 是否需要更新长期文档。

如果某个任务会改变架构、硬件路线、RTOS/框架选择、目录结构或长期协作方式，应先讨论并记录决策。

## Implementation Rules

- 只改本次任务需要的文件。
- 业务代码修改前先确认影响面。
- 不做无关格式化、无关重构或批量整理。
- 不批量删除文件或目录。
- 现有模拟器代码是预研资产，除非任务明确要求，否则不要重写或清理。
- 修改后尽量运行验证命令；如果无法验证，要说明原因。

## Review Rules

Review 时优先看：

- 架构边界是否被破坏。
- 是否又退回“多个循环各干各的”的模式。
- UI 是否直接访问硬件或底层驱动。
- 状态机是否分散在多个页面或任务里。
- 数据模型是否清晰。
- 功耗相关状态是否被忽略。

## Documentation Rules

文档不是装饰，而是项目记忆。

需要更新文档的情况：

- 改变技术路线。
- 增加新模块边界。
- 引入新依赖或框架。
- 改变会话协作方式。
- 完成一个阶段复盘。
- 发现之前的判断有误。

文档应简洁、可执行、能指导下一次行动。
