# v0.7c-1 通知已读状态与详情入口收口

日期：2026-05-24

## 目标

本轮只做一件事：

- 把通知从“只有共享列表”推进到“共享列表 + 已读状态 + 列表内详情入口”

本轮不做：

- 通知详情页独立 `PageId`
- 清空策略重设计
- 已读批量操作
- 通知动作按钮
- 手机侧同步 / 去重 / 分组

## 为什么继续这样拆

`v0.7a` 和 `v0.7b` 已经完成了：

- `NotificationSample -> NotificationService -> DataCenter`
- `ScreenOff` 下通知预览与 `Notifications` 壳层联动

但当前仍缺一层很关键的语义：

- 哪条通知已读，哪条未读
- 从预览进入列表时，这条通知算不算已经读过
- 用户真正“打开一条通知”时，系统里有没有明确状态变化

如果这轮同时加入：

- 已读 / 未读
- 清空规则
- 独立详情页
- 返回路径

就会一下子同时碰数据模型、壳层导航、页面栈和清空语义，回归面明显变大。

所以本轮只收：

- 未读通知进入列表
- 点击列表卡片进入详情
- 进入详情时标记已读
- 返回列表后能看出视觉差异

## 当前前后对比

### 改之前

```text
N 热键
-> NotificationService
-> DataCenter
-> Notifications 列表

通知可以进入系统
但没有 read 状态
列表也没有“进入详情”的正式入口
```

### 改之后

```text
N 热键
-> NotificationService
-> DataCenter(read=false)
-> Notifications 列表(未读样式)
-> 点击卡片
-> Notifications 内部详情态(read=true)
-> 返回列表(已读样式)
```

## 本轮设计决定

### 1. 先不新增独立通知详情页

当前 `Notifications` 是一个壳层临时页。

如果现在直接新增独立 `NotificationDetailPage` 并走普通页面栈，
返回路径会从“回到通知列表”变成“回到主页”，和当前壳层语义不一致。

所以本轮先采用：

- `NotificationsPage` 内部维护 `list/detail` 双态

这样可以在不深改状态机的前提下，把“进入详情”这件事先做真。

### 2. 预览进入列表不自动已读

`NotificationWakePreview` 的意义是“通知到达时的临时唤醒预览”，
不是“用户已经正式阅读了这条通知”。

所以本轮约定：

- 预览出现时不标已读
- 从预览跳进 `Notifications` 列表时仍保持未读
- 只有点击列表卡片进入详情时，才标记已读

### 3. 已读状态先只做最小视觉区分

本轮不做复杂徽标或筛选，只做稳定、低风险的视觉差异：

- 未读：更亮的卡片与文字
- 已读：更暗的卡片与文字

这样已经足够支持手工回归和后续扩展。

## 修改范围

### 数据层

- `NotificationItem` 增加 `read`
- `DataCenter` 增加 `mark_notification_read(...)`

### 页面层

- `NotificationsPage`
  - 增加列表态 / 详情态
  - 点击列表卡片进入详情
  - 返回按钮回到列表
  - 详情打开时标记已读

### 保持不变

- `NotificationService`
- `NotificationWakePreview`
- `NotificationToastOverlay`
- 低电量通知路径
- 长续航路径

## 验收标准

### M1. 新通知默认未读

- 按 `N` 注入一条通知
- 进入 `Notifications`
- 这条通知应以未读样式出现

### M2. 预览进入列表仍未读

- 让手表进入 `ScreenOff`
- 按 `N` 进入 `NotificationWakePreview`
- 从预览进入 `Notifications`
- 该通知仍应保持未读

### M3. 点击进入详情后标记已读

- 在 `Notifications` 列表点击该通知
- 进入详情态
- 使用详情页左上返回按钮回到列表
- 该通知视觉状态应变为已读

### M4. 旧路径不坏

- 按 `B`
- 再测一次长续航进入 / 退出基线
- 不应出现通知改动导致的额外回归
