# Magic Watch Architecture Principles

## Top-Down First

先描述系统行为，再写模块代码。

每个新功能先回答：

- 用户或系统要达成什么状态变化？
- 输入事件是什么？
- 哪个状态机或服务拥有这个行为？
- 数据模型是什么？
- UI 只是展示，还是错误地拥有了业务逻辑？

## Preferred v0 Shape

v0 默认采用轻量架构：

- `Platform`: 隔离模拟器和真实硬件。
- `Service`: 承载时间、电池、传感器、存储、通知等领域能力。
- `App`: 管理系统状态、导航、应用生命周期。
- `UI`: 订阅模型并渲染，不直接访问硬件。
- `Common`: 事件、数据模型、基础工具。

当前 `sim/lv_port_pc_vscode` 已经有类似雏形：`Application`、`DataCenter`、`EventBus`、`PageManager`、`AppStateMachine`、`HAL/Platform Simulator`。后续应优先理解并提炼这条主线，而不是立即推倒重来。

## Event-Driven Over Polling

轮询不是绝对禁止，但业务复杂度不能靠更多循环解决。

推荐方向：

- 输入、时间、电池、传感器变化进入事件流。
- 状态机消费事件并产生导航或服务命令。
- 数据中心保存最近模型并通知订阅者。
- RTOS 任务负责调度和 I/O，不负责分散保存业务状态。

## State Ownership

每种状态必须有清晰主人：

- 电源和屏幕状态属于系统状态机。
- 页面栈属于页面管理器。
- 传感器采样策略属于传感器服务。
- 硬件连接和驱动细节属于平台或 BSP。
- 页面只拥有本页面控件生命周期。

如果一个状态被多个模块同时修改，先停下来重新设计。

## Data-Driven UI

UI 不应成为系统真相来源。

推荐：

- 页面创建时订阅必要模型。
- 模型更新时刷新控件。
- 页面退出时释放订阅。
- 页面发出导航或用户意图事件，而不是直接改其他模块内部状态。

避免：

- 页面直接调用 BSP/HAL。
- 页面间互相调用。
- UI 回调里藏复杂业务分支。

## Framework Discipline

暂不直接引入完整 QPC 框架。

理由：

- 当前更需要训练自顶而下设计和模块边界，而不是先承担完整框架复杂度。
- 现有模拟器已经能承载轻量事件驱动预研。
- QPC 的主动对象、事件队列、层级状态机思想可以先作为设计参考。

未来满足以下条件时再重新评估：

- 已有多个独立服务需要严格事件队列隔离。
- 系统状态机复杂到轻量 FSM 难以维护。
- 已完成模拟器端最小闭环，并能解释引入框架带来的收益和成本。

## Low-Power Awareness

低功耗从架构开始，而不是最后补。

任何服务设计都要考虑：

- 是否常驻。
- 是否能按需启动/停止。
- screen off 时行为如何变化。
- sleep/deep sleep 前如何保存状态。
- 唤醒源是什么。
- 唤醒后如何恢复 UI 和服务。

硬件未定时也要保留这些设计约束。

## Minimal Change Principle

每次改动应尽量小、清楚、可验证。

不为了“看起来更干净”做无关重构。大规模整理必须先写计划、讨论影响面，并确认是否值得。
