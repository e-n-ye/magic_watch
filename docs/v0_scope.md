# Magic Watch v0 Scope

## Current Baseline Note

As of 2026-05-14, the project has a first coherent simulator shell baseline.
The clean summary of that baseline lives in:

- `docs/v0_shell_baseline.md`

本文件定义 Magic Watch 第一条可实现闭环的范围。这里的 `v0` 不是“最终产品第一版”，而是“第一版可讲清楚、可验证、可继续演化的交互与架构样机”。

## v0 Goal

在 PC LVGL 模拟器中建立以表盘为中心的手表主交互闭环：

- 表盘主页显示时间和电池等基础信息。
- 表盘左右进入 4 个快速入口页，并可循环回到表盘。
- 表盘上/下分别进入消息页和快捷设置。
- 表冠按下进入应用页，其他页面按下回到表盘。
- 熄屏状态下表冠按下亮屏回到表盘。
- 熄屏由 5 秒超时、翻腕后的姿态判断、遮盖熄屏等策略触发。
- 熄屏状态下支持可配置的亮屏来源和自动熄屏策略原型。
- 普通页面支持 push/pop 返回。
- 设置、消息、快捷入口、应用页先使用 mock 数据与占位功能，重点验证交互骨架。

## Why This Scope

这次用户给出的方向已经比“做个手表”清晰很多，核心不是零散页面，而是围绕表盘形成一套稳定的交互语言。

但也必须明确一条边界：如果把支付、NFC、消息同步、健康监测、天气、音乐控制都当成当前阶段的真实系统能力，我们会再次把精力耗在接口细节和功能堆叠上，反而学不到最想学的“自顶向下设计”。

因此 v0 的目标是：

- 先做真交互骨架。
- 再做假业务内容。
- 最后才逐步把假内容替换成真服务。

## In Scope

### 1. Watchface-Centered Navigation

- 表盘主页。
- 时间/电池数据刷新。
- 表盘左右滑进入快速入口环。
- 左右滑支持循环。
- 从第 4 个快速入口继续滑动时回到表盘。
- 在表盘与快速入口环中，屏幕左边缘右滑遵循主页环流转逻辑，而不是通用返回逻辑。

### 2. Shell Surfaces

- 表盘顶部下滑进入消息页。
- 表盘底部上滑进入快捷设置页。
- 长按快捷设置项进入详细设置页。
- 表冠按下进入应用页。
- 电源菜单当前仅作为模拟器系统壳层预研入口，不绑定最终硬件输入。
- 长按表冠当前阶段仅预留，不绑定最终功能。

### 3. Common Navigation

- 普通页面 push。
- 普通页面 pop。
- 从非主页上下文返回表盘。
- 页面切换动画和基础状态同步。
- 在应用页、消息页、快捷设置页及其他非主页环页面中，屏幕左边缘右滑应返回上级页面。
- 在具备流式内容的页面中，支持触屏上下拖动浏览和表冠旋转浏览。

### 4. Crown Input Prototype

- 抽象 `CrownPress`。
- 抽象 `CrownRotateCW` / `CrownRotateCCW`。
- 在快速入口环中，旋转表冠可在表盘和快速入口之间流转。
- 在列表型页面中，旋转表冠等价于上下遍历列表。
- 在表盘页中，旋转表冠可驱动局部视觉效果或动画参数。
- 在流式页面中，表冠滚动应与触摸滚动共享一致的内容方向和视觉预期。

### 5. Mock Content

- 快速入口页的支付、音乐、健康、天气、睡眠等仅做 UI 壳与 mock 数据。
- 消息页先显示模拟手机通知。
- 应用页先做静态入口或占位入口。
- 设置页先做列表和详情结构，不追求真实设置生效。

### 6. Scroll Physics Prototype

- 在列表页、消息页、设置页、应用页等流式页面中，支持触摸拖动滚动。
- 手指按住拖动时，内容应尽可能跟手移动。
- 快速滑动并松手后，内容应带有惯性继续滚动一小段，再逐渐减速停止。
- 触摸滚动需要有缓冲和减速感，避免“松手即停”。
- 表冠旋转在流式页面中应驱动同一滚动上下文。
- 当前阶段先验证滚动物理体验和交互一致性，不要求最终参数一次定死。

### 7. Display Wake Policy Prototype

- 支持配置“表冠按下亮屏”。
- 支持配置“翻腕亮屏”。
- 翻腕亮屏支持：关闭 / 全天开启 / 按时间段开启。
- 支持配置“单击亮屏”，默认关闭。
- 支持配置超时熄屏时长，默认 5 秒。
- 支持“姿态自动熄屏”原型，在翻腕亮屏场景下依据姿态变化熄屏。
- 支持“遮盖熄屏”开关。
- 支持“持续亮屏”或低功耗静态表盘显示原型。
- 当前阶段这些策略先在模拟器中做状态和交互验证，不承诺真实功耗表现。

## Out Of Scope

- 真实微信支付、支付宝支付集成。
- 真实 NFC 卡模拟或刷卡流程。
- 真实蓝牙手机连接与通知同步链路。
- 真实音乐控制协议。
- 真实心率、血氧、呼吸、睡眠算法。
- 最终低功耗实现。
- 基于真实触控控制器或 IMU 的功耗优化细节。
- 最终硬件选型绑定。
- 完整应用生态。

## Acceptance Criteria

当 v0 完成时，应能在模拟器中验证：

1. 用户能从表盘进入左右 4 个快速入口页，并循环回表盘。
2. 用户能从表盘进入消息页、快捷设置页、应用页和电源菜单。
3. 用户能通过表冠按压从熄屏态亮屏，在表盘态进入应用页，在其他页面返回表盘，并通过表冠旋转完成主页流转与列表浏览。
4. 用户能在主页环中使用左边缘右滑完成主页表面流转，在非主页环页面中使用同一手势返回上级页面。
5. 用户能在流式页面中感受到跟手拖动、快速滑动后的惯性滚动，以及表冠与触摸滚动的一致性。
6. 用户能在设置中看到并切换亮屏来源、超时熄屏、姿态熄屏、遮盖熄屏、持续亮屏等显示策略原型。
7. 普通页面的 push/pop 逻辑与主页十字导航不互相打架。
8. 当前代码结构能清楚解释“谁处理输入、谁拥有主页状态、谁管理页面栈、谁负责 mock 数据、谁管理显示唤醒策略、谁管理滚动物理规则”。

## v0 Deliverables

- 一份主页与系统交互模型文档。
- 一份目标架构骨架文档。
- 模拟器中的第一版交互样机。
- 后续真实服务替换点清单。

## Phase Hint

建议把 v0 再拆成 3 个小阶段：

1. 先打通表盘十字导航和按键/表冠输入模型。
2. 再补滚动物理规则、显示唤醒策略、快速入口页、消息页、快捷设置页、应用页和设置页的 mock UI。
3. 最后再考虑是否抽出 `Service` 层、显示策略模块、滚动策略模块和更细的主页协调模块。
