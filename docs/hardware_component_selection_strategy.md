# 硬件器件选型策略

日期：2026-05-21

## 为什么需要这份文档

前几轮硬件前哨文档把候选对象主要写成了现成开发板或开源手表参考，例如 LILYGO T-Watch S3 Plus、M5Stack CoreS3、PineTime。

这对快速 bring-up 有价值，但它不是完整的硬件选型。真正的硬件选型需要拆开评估主控、屏幕、触摸、IMU、电源管理和电池路径，再用最小系统板和分立模块组合验证。

因此，本项目从现在开始区分两类硬件验证：

- 参考板验证：用现成板快速暴露真实硬件限制。
- 分立器件选型验证：围绕最终硬件路线，选择主控、屏幕、触摸、IMU、PMIC 等器件，再飞线或小板连接验证。

这两类验证服务同一个目标，但不能互相替代。

## 两层验证模型

### 第一层：参考板验证

代表对象：

- LILYGO T-Watch S3 Plus
- M5Stack CoreS3
- PineTime

价值：

- 快速点亮真实屏幕。
- 快速验证 LVGL、触摸、按键、IMU、电源状态读取等链路。
- 快速暴露模拟器 UI 在真实小屏上的问题。
- 快速观察 screen off / wake 的硬件限制。

边界：

- 参考板不等于最终硬件方案。
- 参考板上的主控、屏幕、触摸、IMU、PMIC 组合已经固定，不能证明这是最优组合。
- 参考板验证通过，只能说明“当前架构可以在某类硬件上跑起来”，不能说明最终选型已经完成。

当前结论：

- 用户已拥有 LILYGO T-Watch S3 Plus。
- 因此第一块参考验证板优先使用现成 T-Watch S3 Plus。
- 它的任务是降低未知，不是替项目做最终器件选择。

### 第二层：分立器件选型验证

代表方式：

- 选择一个主控最小系统板。
- 选择独立屏幕 / 触摸模块。
- 选择独立 IMU 模块。
- 选择电源管理、电池充电和电量检测方案。
- 通过飞线、转接板或小型验证 PCB 连接。

价值：

- 独立评估每个关键器件。
- 避免被现成开发板的固定组合锁死。
- 更接近最终硬件设计流程。
- 可以逐步形成可复用的硬件接口约束。

边界：

- 不在第一轮就追求完整外观、结构和量产可制造性。
- 不在主控未定时承诺最终 RTOS、BSP 或功耗数字。
- 飞线验证只证明电气连接、驱动 bring-up 和基本交互链路，不证明最终可靠性。

## 器件选型分解

正式器件选型至少拆成以下几类。

### 主控

主控选型要回答：

- 是否优先 ESP32-S3 快速验证，还是优先低功耗 MCU 路线。
- RAM / Flash 是否能承载 LVGL、图片资源、页面缓存和后续服务。
- 是否有足够 SPI / QSPI / I2C / GPIO / PWM / ADC / USB / SWD 或 JTAG 资源。
- sleep / deep sleep 能力是否和屏幕、触摸、IMU 唤醒策略匹配。
- SDK、调试、日志和社区资料是否足够支撑长期学习。

当前不可直接下结论：

- 不能因为 T-Watch S3 Plus 用 ESP32-S3，就默认最终主控也是 ESP32-S3。
- 也不能因为低功耗目标存在，就直接跳过 ESP32-S3 的快速验证价值。

### 屏幕与显示驱动

屏幕选型要回答：

- 尺寸是 240 x 240、240 x 296，还是其他比例。
- 屏幕形态是方形、圆角矩形、圆形，还是其他。
- 显示类型是 LCD、AMOLED、Memory LCD、e-paper，还是其他。
- 显示接口是 SPI、QSPI、RGB、MIPI 或并口。
- 是否支持局部刷新、低刷新率、低功耗静态显示或 AOD。
- LVGL 帧缓冲和刷新性能是否可接受。

当前重点：

- T-Watch S3 Plus 的 240 x 240 屏幕可用于验证真实小屏压力。
- 当前模拟器的 240 x 296 方向仍应保留为设计假设之一，不能因为 T-Watch 固定为 240 x 240 就立即推翻。

### 触摸控制器

触摸选型要回答：

- 是否支持多点触摸。
- 是否支持中断输出。
- 是否支持低功耗触摸唤醒扫描。
- 是否能稳定识别点击、拖动、滑动和边缘手势。
- 驱动资料是否完整。

当前重点：

- T-Watch S3 Plus 的 FT6336U 可作为触摸参考验证对象。
- 但 `TapToWake` 是否真实可行，必须看触摸控制器和电源域是否支持低功耗唤醒。

### IMU / 姿态传感器

IMU 选型要回答：

- 是否支持低功耗运动中断。
- 是否支持抬腕、敲击、方向变化或自定义阈值中断。
- 主控睡眠时 IMU 中断是否仍可唤醒系统。
- 驱动和配置资料是否容易获得。
- 姿态算法是先用芯片内置特性，还是后续自研。

当前重点：

- T-Watch S3 Plus 的 BMA423 可作为 `RaiseToWake` 参考验证对象。
- 但不能因为 BMA423 可用，就默认最终 IMU 选 BMA423。

### 电源管理、充电和电量检测

电源选型要回答：

- 是否需要独立 PMIC。
- 是否需要独立充电管理。
- 是否需要 fuel gauge，还是用 ADC 估算电池电压。
- 屏幕、触摸、IMU、无线、震动等外设是否需要分域供电。
- sleep / deep sleep 后哪些状态保持，哪些必须重建。
- 是否容易测量功耗。

当前重点：

- T-Watch S3 Plus 的 AXP2101 可作为 PMIC 参考验证对象。
- 但最终电源方案仍需要独立评估，不应直接继承参考板设计。

## 当前推荐推进顺序

### 第一步：用 T-Watch S3 Plus 做参考板核验

目标：

- 不改最终选型。
- 不飞线。
- 不拆硬件。
- 先验证真实手表开发板会暴露哪些问题。

核验内容：

- 屏幕点亮和最小 LVGL。
- 触摸点击 / 滑动基础事件。
- BMA423 姿态数据或中断路径。
- AXP2101 电池 / 充电 / 电源状态读取。
- screen off / wake 后 UI 状态是否还能恢复。
- 240 x 240 屏幕对当前 240 x 296 模拟器假设的压力。

输出：

- 一份 T-Watch S3 Plus 参考板核验记录。
- 明确哪些问题来自“真实硬件共性”，哪些只是 T-Watch 板级限制。

### 第二步：建立分立器件选型表

目标：

- 拆主控、屏幕、触摸、IMU、电源管理。
- 不立即购买模块。
- 先列评价维度和候选类别。

输出：

- 主控候选表。
- 屏幕 / 触摸候选表。
- IMU 候选表。
- PMIC / 充电 / 电量检测候选表。

### 第三步：选择第一套飞线验证组合

目标：

- 选择一个主控最小系统板。
- 选择一个屏幕 / 触摸模块。
- 选择一个 IMU 模块。
- 选择最小电源 / 电池检测方案。

原则：

- 第一套组合优先验证“能跑通”，不是最终小型化。
- 模块资料和引脚可访问性优先。
- 不同时引入过多高风险器件。

输出：

- 飞线连接计划。
- bring-up 顺序。
- 失败时的替代路径。

## 当前阶段的正确用法

T-Watch S3 Plus 的正确用法：

- 作为真实手表参考板。
- 作为架构假设核验工具。
- 作为触摸、IMU、PMIC、屏幕链路的观察对象。
- 作为后续分立器件选型的对照组。

T-Watch S3 Plus 的错误用法：

- 直接宣布 ESP32-S3 是最终主控。
- 直接宣布 240 x 240 是最终屏幕规格。
- 直接宣布 FT6336U、BMA423、AXP2101 是最终器件。
- 为了适配 T-Watch 立即重写模拟器 UI 和状态机。
