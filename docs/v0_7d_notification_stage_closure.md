# v0.7d 通知阶段收口

日期：2026-05-24

## 目标

本轮不新增通知功能，只做一件事：

- 把 `v0.7a` 到 `v0.7c-2b-1` 的通知工作收口成一个明确阶段结果

本轮不做：

- 新的通知交互
- 详情页独立化
- 通知删除手感细化
- 手机侧同步

## 为什么现在先收口

通知这条线已经连续推进了多轮：

- `v0.7a`：通知 service 最小闭环
- `v0.7b`：通知预览与壳层联动
- `v0.7c-1`：已读状态与列表内详情入口
- `v0.7c-2a`：清空确认
- `v0.7c-2b-1`：单条右滑删除

而且在这个过程中还修掉了一个真实 bug：

- 从通知详情页上滑退出后，重新下拉错误地回到旧详情态

这说明通知线已经从“缺骨架”进入“继续抠会逐步收益递减”的阶段。

现在最有价值的不是继续加通知功能，而是：

- 明确通知线当前已经具备什么
- 明确还故意没做什么
- 给下一阶段让出空间

## 当前通知链路的收口结果

### 1. 上游生产已经成型

当前通知的进入路径已经是：

```text
N 热键
-> SimulatorDevice(NotificationSample)
-> NotificationService
-> DataCenter
-> Notifications / WakePreview / Toast
```

这意味着：

- 通知已经不再是 `Application` 里直接塞 mock 数据
- “通知如何进入系统”已经有明确 service 边界

### 2. 壳层路径已经稳定

当前通知壳层的主路径是：

- `ScreenOff + 通知亮屏允许` -> `NotificationWakePreview`
- `Running + 不在通知列表` -> toast
- `Notifications` 已打开 -> 只刷新列表

并且：

- 从 `NotificationWakePreview` 可以进入完整 `Notifications`
- 关闭通知壳层后，不再保留旧详情态

### 3. 通知列表已经具备最小真实交互

当前列表已经支持：

- 未读 / 已读
- 点击进入详情
- 详情内返回列表
- 清空确认
- 单条右滑删除

也就是说，通知列表已经不再只是一个“摆几条内容的壳子”，而是一个最小可交互对象。

## 当前故意保留未做的内容

这些不是“忘了做”，而是当前阶段有意不做：

- 独立 `NotificationDetailPage`
- 只清已读
- 删除背景提示
- 撤销删除
- 通知动作按钮
- 手机侧同步 / BLE / 权限
- 通知分组、去重、优先级

原因很一致：

- 继续往下做会明显扩大状态机、页面栈和产品规则的耦合面
- 当前项目的更高优先级，已经不再是通知，而是下一条共享模型边界

## 本阶段最重要的经验

通知线给项目带来的价值，不是“通知页更像真机了”这么简单，而是再一次验证了这条主线：

```text
平台样本
-> service
-> DataCenter
-> 壳层 / 页面多个消费者
```

前面我们已经用它做过：

- Battery
- Steps / Activity

现在通知也落到了同一种结构里。

这对项目真正重要，因为它说明：

- 系统骨架不是只适用于某一类数据
- 它已经能承载多种对象类型

## 阶段验收结论

当前通知阶段可以视为“稳定封箱”的条件是：

- `N` 注入通知正常
- `WakePreview -> Notifications` 正常
- 未读 -> 详情 -> 已读正常
- 清空确认正常
- 单条右滑删除正常
- 从详情页上滑退出后，重新进入通知回到列表态
- `B` 低电量通知不坏
- 长续航路径不坏

如果以上都通过，就不建议继续优先扩通知。

## 下一阶段建议

通知收口后，下一步最值得进入的是：

- `v0.8 健康监测设置模型闭环`

原因：

- 它能把“续航优化页里的多个健康开关”从页面局部状态，推进成共享模型
- 它比继续抠通知更符合当前项目“复杂系统骨架训练”的目标
- 它又不会像真传感器接入那样，太早把主线拖回硬件细节

## 建议的切换方式

下一轮不应该直接做心率/血氧真逻辑，而应先做：

1. 文档化健康监测设置模型的边界
2. 明确哪些开关属于同一个共享对象
3. 明确这些开关未来会被哪些页面消费
4. 再做最小代码闭环

先把“谁拥有这组状态”讲清楚，再进入实现。
